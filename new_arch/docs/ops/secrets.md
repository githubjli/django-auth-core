# Secrets Management

Secrets never live in the repo. Not in code, not in `.env` files, not in test fixtures.

---

## 1. What counts as a secret

| Secret | Notes |
|---|---|
| JWT private key (Identity) | most sensitive in the system |
| Database passwords | per-service, rotated quarterly |
| Redis passwords | if enabled in V2 |
| Stripe live secret key | per-environment |
| Stripe webhook secret | per-environment |
| SendGrid / Twilio / FCM API keys | per-environment |
| Ant Media admin credentials | V3 |
| Blockchain node access tokens | per-network |
| Cloud provider credentials (S3, DNS, etc.) | per-environment |
| OAuth client secrets | if implemented |
| Doppler / Secrets Manager access tokens | bootstrap secret |

**Not** secrets (live in env files / config maps in cleartext):
- Service hostnames / ports
- Public keys / JWKS URLs
- Feature flag identifiers
- Non-sensitive timeouts and limits
- `ALLOWED_HOSTS`, `DJANGO_SETTINGS_MODULE`

---

## 2. Tool choice

**V1 recommendation: Doppler.**
Alternative: AWS Secrets Manager if already on AWS.

Decision recorded in ADR-0010 supplement (extend ADR if changing).

Pattern:
- Secrets stored in the chosen manager
- Application reads secrets at startup via secrets manager SDK or pre-mounted files
- Secrets are mounted as **files** at known paths (`/run/secrets/jwt-private.pem`), not env vars where possible — env vars are visible in `ps` and crash dumps

## 3. Local development

Two layers:

### Layer 1 — Generic dev secrets
- Provided in `.env.example` (committed, no real values)
- Each developer copies to `.env.local` (git-ignored)
- Loaded via `direnv` or `python-dotenv`

### Layer 2 — No real secrets needed for most work
- Default stack (`make dev`) uses ephemeral local credentials
- Stripe test mode
- Local JWT keypair generated by `make dev-keys` (git-ignored)

### `.env.example` template
```bash
# brandable-content-platform local dev environment
# Copy to .env.local and customize. NEVER commit .env.local.

ENVIRONMENT=local
DJANGO_SETTINGS_MODULE=config.settings.local

# Database (docker-compose default)
DATABASE_URL=postgresql://bcp:bcp@localhost:5432/bcp_dev

# Redis (docker-compose default)
REDIS_URL=redis://localhost:6379/0

# JWT (generated locally by `make dev-keys`)
JWT_PRIVATE_KEY_PATH=./ops/dev-keys/jwt-private.pem
JWT_PUBLIC_KEY_URL=http://localhost:8000/.well-known/jwks.json

# gRPC services (local)
NOTIFICATION_SERVICE_ADDR=localhost:50051

# Stripe (test keys, get yours from Stripe dashboard)
STRIPE_PUBLISHABLE_KEY=pk_test_...
STRIPE_SECRET_KEY=sk_test_...
STRIPE_WEBHOOK_SECRET=whsec_...

# Blockchain (devnet / testnet)
BLOCKCHAIN_LBC_NODE_URL=http://localhost:5279
BLOCKCHAIN_LTT_NODE_URL=https://testnet.ltt.example.com

# Email (console backend)
EMAIL_BACKEND=django.core.mail.backends.console.EmailBackend

# Telemetry (optional locally)
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318
```

`ops/dev-keys/` is generated by `make dev-keys` on first run, git-ignored, only used locally.

## 4. CI / CD

GitHub Actions: secrets in repo Settings → Secrets and variables → Actions. Scope per environment (`staging`, `production`).

Ansible reads secrets via lookup plugin against Doppler / Secrets Manager during playbook run. No secret material in Ansible vars files.

---

## 5. JWT private key (highest-sensitivity)

- Stored in secrets manager only
- Identity service is the only consumer
- Mounted at `/run/secrets/jwt-private.pem` with mode 0400, owner = service user
- Public key distributed via JWKS endpoint at `/.well-known/jwks.json` — public-readable
- Rotation: see `runbooks/jwt-key-rotation.md`

---

## 6. Database credentials

- Per service: Django has `bcp_django` user, NotificationService has `bcp_notification` user (own DB and own credentials)
- Migration scripts use a separate `bcp_migrator` user with DDL grants
- Read-only analytics user `bcp_readonly` for ad-hoc queries (V2)
- Credentials rotated every 90 days minimum, automated via Doppler rotation policies

---

## 7. Third-party API keys

- Each environment has its own set (dev / staging / production)
- Stripe, SendGrid, etc. all support separate test and live keys — never mix
- Rotation triggered:
  - On team member departure with key access
  - On suspected leak (gitleaks detected, etc.)
  - On schedule (annual minimum)

---

## 8. Detection

- `gitleaks` runs in pre-commit AND CI; anything matching secret patterns blocks the commit
- Periodic full-repo scan in CI (weekly)
- If a secret leaks: **rotate immediately**, then audit who accessed when

---

## 9. Audit

- Every production secret access logs to the secrets manager's audit trail
- Quarterly review: who accessed what, what's stale, what should be rotated
- Self-audit: secret manager access by admin user produces `identity.session.revoke` + `audit.export` log entries (per audit.md §5)

---

## 10. Anti-patterns

- ❌ `print(api_key)` for "debugging"
- ❌ Committing `.env`
- ❌ Sharing secrets via Slack DM
- ❌ Storing secrets in Notion / Confluence
- ❌ Reusing the same key across environments
- ❌ Leaving rotation to "later"
- ❌ Mounting secrets as env vars when files are an option
- ❌ Hardcoding rotation passwords in playbooks
